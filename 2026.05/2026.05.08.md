## 2. 프로그래머스 문제 풀이

### [공원 산책](https://school.programmers.co.kr/learn/courses/30/lessons/172928)

```cpp
#include <string>
#include <vector>
#include <algorithm>

using namespace std;

vector<string> solution(vector<string> strings, int n) {
    sort(strings.begin(), strings.end(), [&n](const string& A, const string& B){
        if(A[n] != B[n]) return A[n] < B[n];
        else return A < B;
    });
    return strings;
}
```

* **문자열** 문제
* 문제의 조건을 `sort()` 함수의 람다로 구현

### [달리기 경주](https://school.programmers.co.kr/learn/courses/30/lessons/178871)

```cpp
#include <string>
#include <vector>
#include <unordered_map>
using namespace std;

vector<string> solution(vector<string> players, vector<string> callings) {
    unordered_map<string, int> rank;
    
    for(int i=0; i<players.size(); i++)
        rank[players[i]] = i;

    for(const string& calling : callings){
        int index = rank[calling];
        players[index] = players[index-1];
        players[index-1] = calling;
        rank[calling]--;
        rank[players[index]]++;
    }
    return players;
}
```

* **해시맵**을 활용한 문제
* 해시맵을 통해 불린 선수의 순위를 찾음
* 바로 앞의 선수와 위치를 바꿔줌

## 3. 스택 메모리

* `OS`는 메모리를 크게 Code, Data, Stack, Heap의 네 가지 영역으로 나눔 

* 이중 **스택** 영역은 프로그램이 **자동으로 사용**하는 임시 메모리 영역

* **스택** 영역은 **함수의 호출**과 관련된 데이터들이 저장되며 이를 **스택 프레임(Stack Frame)** 이라고 함

* **스택 프레임**은 함수가 종료되면 자동으로 삭제 됨

### 스택 오버플로우 (Stack Overflow)

* **오버플로우**란 한정된 메모리 공간이 부족하여 **다른 메모리 공간을 침범**하는 것

* 너무 많은 함수가 실행 중일 경우 **스택 프레임**의 크기가 **스택의 전체 크기보다 커져** 발생할 수 있음

### 스택 오버플로우의 주요 발생 원인

* **무한 재귀 호출 (Infinite Recursion):** 재귀 함수는 **종료 조건**을 명확하게 설정하지 않으면 무한히 재귀가 발생할 수 있음

* **깊은 재귀 (Deep Recursion):** **피보나치 수열**과 같이 종료 조건이 명확하더라도 **재귀의 깊이가 너무 깊으면** 스택의 크기보다 커질 수 있음

* **과도한 지역 변수 선언 (Large Stack Allocation):** 크기가 매우 큰 배열과 같이 매우 큰 **지역 변수**를 선언하면 스택 메모리를 초과할 수 있음

### 해결 방법

* **꼬리 재귀 최적화 (Tail Recursion Optimization):** 재귀 함수의 **마지막 연산이 재귀 함수 호출**일 경우, **컴파일러가 반복문으로 바꾸어 실행**하는 것

* **메모이제이션 (Memoization):** **동일한 계산**을 반복하는 경우, 해당 값을 **저장**해두어 함수를 호출하는 것이 아닌 값을 읽는 것

* **동적 할당 (Dynamic Allocation):** 일반적으로 **힙 영역**은 **스택 영역**보다 넉넉하므로 크기가 큰 변수는 동적할당을 하는 것이 안전

* 스택의 크기를 조정하는 방법이 있으나, 이는 근본적인 해결 방법은 아님
