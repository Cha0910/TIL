## 2. OOP

### OOP란?
* `Object Oriented Programming`의 약자로 한국어로 하면 `객체 지향 프로그래밍`
* **객체 지향 프로그래밍**은 컴퓨터 프로그래밍의 **패러다임** 중 하나로 프로그램을 **객체의 집합**으로 보는 것
* 각각의 객체가 유기적인 **상호 작용**을 통해 프로그램의 시스템을 구성하는 것을 의미함


### OOP의 기본 구성 요소
* **클래스(Class)** 
    * 어떠한 일을 수행하기 위하여 **속성**(attribute)과 **행위**(behavior)를 정의한 것 
    * OOP에서는 **사용자 정의 데이터 타입** 이라고 할 수 있음
    * 흔히 객체를 생성하기 위한 **틀**에 비유함

* **객체(Object)**
    * 실제로 **메모리에 할당** 된 클래스의 **인스턴스**
    * 클래스에서 정의한 **속성**을 가지고 **행위**를 직접 수행할 수 있음

* **메소드(Method)**
    * 생성된 객체를 사용하는 방법으로, 객체에 **명령을 내리는 것**이라 할 수 있음
    * 흔히 클래스 내부에 선언된 **함수**를 의미하며, 객체의 **속성을 조작**하거나 **다른 객체와 상호작용** 하는데 사용

## 3. OOP의 4대 특징

### 추상화 (Abstration)
* **추상**의 사전적인 의미는 '여러 가지 사물이나 개념에서 공통되는 특성이나 속성 따위를 추출하여 파악하는 작용'
* **OOP** 에서의 **추상화** 또한 사전적인 의미와 유사한데, 핵심은 **공통적인 특성**, **속성**만을 추출하여 **하나의 객체**로 만드는 것
* 이를 통하여 **불필요한 정보는 숨기고**, **중요한 정보만을 표현**해 프로그램을 간단히 만드는 것을 목표로 함

### 상속 (Inheritance)
* 새로운 클래스가 기존 클래스의 내용을 **물려받는 것**
* 이를 통해 자식 클래스는 부모 클래스의 **속성과 행위를 가져 사용**할 수 있으며, 새로운 속성과 행위를 추가하여 **기능을 추가**할 수 있음

### 다형성 (Polymorphism)
* 상속된 어떤 인터페이스나 메소드를 **상황에 따라서 다르게 동작**하게 하는 것
* 대표적인 예시로 매개 변수를 다르게 하는 **오버로딩**과 자식 클래스에서 메소드를 재정의하는 **오버라이딩**이 존재함

### 캡슐화 (Encapsulation)
* 클래스 내부의 속성과 행위를 하나의 캡슐로 만들어 **외부로부터 보호**하는 것
* 객체 내부의 세부 내용이나, 데이터를 외부로 부터 숨기는 **정보 은닉**을 목표로 함
* **접근 제어자**를 통해 외부의 접근을 차단, 허용할 수 있음

## 4. OOP의 5대 설계 원칙 (SOLID)

### 단일 책임 원칙 - SRP (Single Responsibility Principle)
* **`한 클래스는 하나의 책임만 가져야 한다`**
* 즉 하나의 클래스는 **하나의 기능**만을 담당해야 한다는 것을 의미함
* **기능 변경**이 일어났을 때, **다른 클래스 또한 수정**이 필요해 진다면 이는 **단일 책임 원칙**을 준수하지 않은 것 
* 단일 책임 원칙은 코드의 **가독성 향상**, **유지 보수**를 더 용이하게 해줌

### 개방 폐쇄 원칙 - OCP (Open Closed Principle)
* **`소프트웨어 요소는 확장에는 열려 있으나 변경에는 닫혀 있어야 한다`**
* 확장이란, **새로운 기능이 추가**되는 것을 의미함
* 변경이란, **기존의 코드를 수정** 하는 것을 의미함
* 즉 **계방 폐쇄 원칙**은 기능 추가 시 클래스를 간단하게 확장하고 수정은 최소화 하는것을 목표로 함
* 객체를 **추상화** 하는 것이 **개방 폐쇄 원칙**을 준수하는 것

### 리스코프 치환 원칙 - LSP (Liskov Substitution Principle)
* **`프로그램의 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 한다`**
* 즉, 자식 클래스는 최소한 부모 클래스의 메소드에 대해서 **의도대로 수행이 보장**되어야 함
* **다형성**을 사용할 때, 반드시 지켜야 하는 원칙
* 쉽게 말하면 **업캐스팅** 된 자식 객체에서 부모의 함수를 실행 했을 때, 의도한대로 작동해야하는 원칙

### 인터페이스 분리 원칙 - ISP (Interface Segregation Principle)
* **`특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 낫다`**
* **범용적인 인터페이스**보다 특정 기능을 담당하는 **인터페이스로 분할**해야 한다는 원칙
* 인터페이스를 범용적으로 만들 경우, 어떤 클래스는 **쓰지도 않을 메소드를 작성**해야만 하는 경우가 생김
* **인터페이스 분리 원칙**은 **인터페이스의 단일 책임 원칙**을 의미함

### 의존 역전 원칙 - DIP (Dependency Inversion Principle)
* **`추상화에 의존해야지, 구체화에 의존하면 안된다`**
* 어떤 클래스를 참조할 경우, 그 클래스를 직접적으로 참조하지 않고 **추상 클래스나 인터페이스를 참조**하라는 원칙
* 하위 클래스를 직접적으로 참조하면 **결합도**가 높아져 하위 클래스에서 수정이 발생할 경우, **코드의 수정이 빈번해짐**
