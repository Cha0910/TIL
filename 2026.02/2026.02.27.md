## 3. 인벤토리 및 장착 시스템

### 인벤토리 구현

* 캐릭터의 인벤토리는 캐릭터에 기능을 추가하지만, 게임 월드에는 존재하지 않아야 하기에 `액터 컴포넌트(Actor Components)` 클래스를 사용하여 정의
* 인벤토리는 포인터 배열 `TArray<UEquippableToolDefinition*> ToolInventory`을 통해 다양한 아이템을 관리하게 끔 구현

### FAttachementTransformRules

* 스폰된 툴을 캐릭터에 붙이기 위해 새로운 `FAttachementTransformRules`을 선언해야함
* `FAttachementTransformRules`는 부착 시 위치와 회전, 스케일 조절 처리 방법을 정의하는 `구조체`이며 아래 두 개의 파라미터를 받음
  * **EAttachmentRules:** 자식 액터가 부모 액터에 부착 될 때 **어떻게 부착될지 결정하는 규칙**이며 주로 아래 3개의 규칙을 사용함

    | 규칙 (Rule)  | 설명 | 
    | --- | --- |  
    | **SnapToTarget**  | 자식을 부모(또는 소켓) 위치로 **고정**
    | **KeepRelative**  | 부모를 기준으로 자식이 가진 **상대적인 거리**를 유지 | 
    | **KeepWorld**  | 부착되는 순간 자식이 서 있던 **월드 좌표**를 유지 | 

  * **InWeldSimulatedBodies:** 두 액터의 `Physics Bodies`를 하나로 합칠 것인가를 결정하는 `bool` 값

### 다트 발사기 구현 중 트러블 슈팅

* **현상:** 다트 발사기의 `Use()`가 입력 키를 눌러도 실행되지 않음
* **원인:** `UE_LOG`를 통해 `void ADartLauncher::BindInputAction(const UInputAction* InputToBind)`의 **InputToBind** 인자가 `nullptr`을 참조함을 확인
* **해결:** 캐릭터 블루프린트의 해당 프로퍼티 칸에 `IA_UseTool`을 할당

## 4. Enhanced Input
* 위 문제는 단순한 휴먼 에러였고 `Enhanced Input System`의 이해가 부족해 발생한 문제라고 생각하여 추가로 학습함

### 입력 액션(Input Actions)
* Enhanced Input System과 코드 간의 통신 링크 사용자의 입력에 이름을 붙여 프로그램 내에서 쉽게 참조 및 관리하게 해줌

### 입력 매핑 컨텍스트 (Input Mapping Contexts)
* 사용자의 실제 입력이 어떤 입력 액션과 대응하는지 정의하는 컨텍스트
* 하나의 캐릭터에 상태에 따라 다른 컨텍스트 변경도 가능하며, 여러 개의 컨텍스트 부여도 가능
* 다수의 컨텍스트가 동일한 키를 매핑하고 있으면 가장 높은 우선순위의 입력 액션을 실행

### 입력 모디파이어(Input Modifiers)
* 사용자의 실제 입력값을 입력 트리거로 보내기 이전에 변경, 조정하는 단계
* 코드 수정 없이 `값 반전(Negate)`, `축 변경(Swizzle)` 등을 조정 가능

### 입력 트리거(Input Triggers) 
* 입력 모디파이어로부터 조정된 사용자의 입력을 받아 입력 매핑 컨텍스트의 해당 입력 액션을 `실제로 실행할지 여부를 결정`하는 단계
* 입력 트리거의 유형은 아래의 세 가지 유형이 있음

  | 유형 | 논리 기호 | 판정 방식 | 
  | --- | --- | --- | 
  | **명시 (Explicit)** | **OR** | 여러 명시 트리거 중 **하나라도** 성공하면 액션 실행 | 
  | **암시 (Implicit)** | **AND** | 모든 암시 트리거가 **전부** 성공해야 액션 실행 | 
  | **블로커 (Blocker)** | **NOT** | 블로커 트리거가 성공하면 **무조건 액션 차단** |
